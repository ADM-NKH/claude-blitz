# Blitz v0.2 Design — Compounding Background Work

Date: 2026-05-04
Status: Draft, awaiting review

## Problem

Blitz v0.1 only fires before quota resets. To make use of more of an active week — across all the hours when the user is away from the keyboard — blitz needs to (a) fire across the whole week, not just the last 45 minutes before a reset, and (b) have something useful to run when the user hasn't queued anything.

The v0.1 pivot away from `max-mode` set two principles in stone:

- **Real work only.** No invented filler to "burn quota." Parallelism is throughput, not a multiplier.
- **Persist everything.** Every fire writes artifacts the user can act on.

v0.2 must extend reach without breaking either principle.

## Non-goals

- Multiplying quota. v0.2 does not bypass, parallelize-for-multiplication, or otherwise abuse plan limits.
- Pre-reset cramming. The "fire heavier batch right before reset" idea was rejected during design — it re-introduces the burn-quota framing.
- Detecting Anthropic-side usage numbers (no Playwright on claude.ai, no scraping of plan dashboards).
- Replacing manual `/blitz`. The on-demand backlog runner stays.

## Architecture

### Unified backlog with item kinds

v0.1's `backlog[]` becomes a single queue with a `kind` field:

```json
{ "id": 1, "kind": "task" | "goal" | "audit", "task": "...", "project": "...", ...kind-specific-fields }
```

- `kind: "task"` — a unit of work (current v0.1 backlog item, unchanged behavior)
- `kind: "goal"` — a long-term goal that spawns task items via a durable plan
- `kind: "audit"` — a maintenance audit configured for a project

This is additive: v0.1 backlog items get `kind: "task"` defaulted on read, no destructive migration.

### Three work sources, one queue

- **Goals** are user-defined, decompose at run-time from a durable plan, and emit increments on a long-lived branch per goal.
- **Tasks** are the v0.1 backlog: explicit, user-queued, run-and-remove.
- **Audits** are project-level maintenance jobs that produce read-only reports and auto-promote one finding per run into a task.

### Pull priority

When an auto-fire runs, it walks item kinds in this order until it has filled the fire's capacity:

1. Goals (max 1 per fire)
2. Tasks (up to 3 per fire)
3. Audits (max 1 per fire)

Caps are configurable. Per-project idle gating may cause individual items to skip; if every pulled item skips, the fire logs "all targets active, deferring" and exits clean.

### Empty-state behavior

If goals, tasks, and audit-enabled projects are all empty, the fire writes a one-line log entry and exits silently. **Auto-fire never invents work.** This is identical to v0.1's "empty backlog = no run" rule, just extended across the new sources.

## Goal lifecycle

### Storage

Each `kind: "goal"` item has:

```json
{
  "id": 5,
  "kind": "goal",
  "task": "Ship pluginproof MVP",
  "project": "C:\\Users\\Adam\\MCP\\pluginproof",
  "added": "2026-05-04",
  "branch": "blitz/goal-5",
  "autoPush": false,
  "lastTouched": "2026-05-04T14:00:00Z",
  "incrementCount": 0
}
```

The durable plan lives on disk, **not in the config file**:

- Path: `~/blitz/goals/<id>/plan.md`
- Format: numbered Markdown checklist with one increment per line
- Source of truth for "what is the next thing to do for this goal"

### First fire on a goal (planning fire)

When a goal has no `plan.md` (or empty plan.md), the goal-fire agent does this and only this:

1. Read the goal text and project state (`git log -30`, file tree summary, `README.md` if present).
2. Decompose into a numbered checklist of small, mergeable increments. Each item must be self-contained and demoable.
3. Write the checklist to `~/blitz/goals/<id>/plan.md`.
4. Write a result file to `~/blitz/runs/<ts>/goal-<id>-plan.md` summarizing the plan and the reasoning.
5. Exit. **No code commits on the first fire.** This prevents stomping and gives the user a chance to edit the plan before any code is written.

### Subsequent fires on a goal (increment fires)

1. Load `plan.md`. If every item is checked, mark the goal completed and exit.
2. Pick the lowest unchecked item.
3. Switch to / create the long-lived `blitz/goal-<id>` branch (rebased on `main`).
4. Implement that item. Run tests if the project has them and they're cheap.
5. Commit on `blitz/goal-<id>`. Do not push unless `autoPush: true` is set explicitly per-goal.
6. Update `plan.md`: mark the item done, append a one-line summary with the commit SHA.
7. Write a result file to `~/blitz/runs/<ts>/goal-<id>-<slug>.md` with: what was done, why, what's next, branch/commit reference.
8. Increment `incrementCount`, update `lastTouched`.

### Goal review

`/blitz goal review <id>` runs:

```
git log <branch> ^main --oneline
git diff main..<branch> --stat
```

…and prints the plan.md status (checked / unchecked items). User decides when to merge or rebase.

### Goal commands

| Command | Action |
|---|---|
| `/blitz goal add <text>` | Add a goal (cwd or `--project`). Creates the item; plan.md is written by the first fire, not at queue-time. |
| `/blitz goal list` | List goals with branch, increment count, last-touched date. |
| `/blitz goal review <id>` | Show the goal's branch diff vs main and plan.md status. |
| `/blitz goal log <id>` | Print increment summaries from result files. |
| `/blitz goal remove <id>` | Remove the goal from the backlog. Does NOT delete the branch or plan.md (user cleans up). |
| `/blitz goal autopush <id> on/off` | Toggle whether increment fires also push to origin. |

## Audit lifecycle

### Storage

Audits are stored in the same backlog with `kind: "audit"`:

```json
{
  "id": 12,
  "kind": "audit",
  "project": "C:\\Adam\\claude_skills",
  "rotation": ["security", "deadcode", "tests", "deps", "docs", "todos"],
  "rotationIdx": 0,
  "lastSwept": "2026-05-01T08:00:00Z",
  "promotedFindings": ["sha256:abc123...", "sha256:def456..."]
}
```

`promotedFindings` stores the SHA-256 hashes of findings already promoted, used as an idempotency key so the same TODO/issue isn't re-queued every audit.

### Audit types and outputs

| Audit | Output |
|---|---|
| `security` | OWASP-style issues with file:line + severity |
| `deadcode` | Unused exports, unreachable branches, orphaned files |
| `tests` | Public functions with no test coverage |
| `deps` | Outdated packages, breaking-change risk flagged |
| `docs` | Public APIs without docstrings/JSDoc |
| `todos` | TODOs/FIXMEs grouped by age and severity |

### Audit fire flow

1. Pick the next audit type from `rotation` (cycles through; `rotationIdx` increments after each fire).
2. Run the audit. Output is a markdown file at `~/blitz/runs/<ts>/audit-<project>-<type>.md`.
3. From the findings, pick the highest-severity one whose hash is **not** in `promotedFindings`.
4. Auto-promote that finding as a `kind: "task"` backlog item with the original audit-id and finding-hash recorded.
5. Add the finding's hash to `promotedFindings`.
6. Update `lastSwept` and `rotationIdx`.
7. **Audits never modify project files.** Reports go to the output dir; promoted findings go to the backlog. The user merges/discards.

### Audit commands

| Command | Action |
|---|---|
| `/blitz audit enable <project>` | Add project to audit rotation. Creates a `kind: "audit"` item with the default rotation. |
| `/blitz audit disable <project>` | Remove the audit item for that project. |
| `/blitz audit list` | Show enabled projects, last-swept dates, next audit type per project. |
| `/blitz audit run <project>` | Run the next audit immediately (manual trigger). |

## Firing model

### Hybrid cadence + per-project idle gate

Replaces v0.1's "fire 45 min before each reset."

- **Cadence** (`cadenceHours`, default 3): interval between fire attempts, anchored at config-time.
- **Per-target idle gate**: for each item the fire intends to run, check the item's `project`:
  - Skip the item if any commit landed in the project in the last `idleMinutes` (default 30) — checks `git log --since`.
  - Skip the item if `git status --porcelain` returns non-empty (working tree dirty).
- If every pulled item skips, the fire logs `all targets active, deferring` and exits clean. The next fire happens on schedule; no immediate retry.
- **Optional blackout windows**: an array of weekday + start/end ranges where firing is suppressed entirely (e.g., user's typical work hours).

### What was rejected

- **Pre-reset boost.** Earlier draft included a "fire a heavier batch within preResetMinutes of weekly reset" rule. Removed during review: this is exactly the burn-quota framing v0.1 rejected. Manual `/blitz` covers explicit pre-reset runs.
- **Global jsonl mtime as idle signal.** Reading `~/.claude/projects/*/<session>.jsonl` mtime fires false positives (any unrelated session blocks the gate) and false negatives (user editing in VS Code without a Claude session leaves the gate open). Per-project git-based detection is more accurate.
- **Anthropic-side quota detection.** Out of scope. TOS-risky and fragile.

### Kill switches (preserved from v0.1)

- `/blitz off` — disable all auto-fires
- `/blitz on` — re-enable
- `/blitz skip` — skip the next scheduled fire only
- All scheduled-task / cron entries can be removed by `/blitz off` if the user wants OS-level disable.

## Migration v1 → v2

Detected when reading config and version is `1` or absent.

1. Add `kind: "task"` to every existing backlog item.
2. Replace `weeklyReset`-only firing with the new `firing` block. Defaults: `mode: "hybrid"`, `cadenceHours: 3`, `idleMinutes: 30`, no blackouts.
3. Drop `preResetMinutes` (was in v1 schema; functionality removed).
4. Drop `sessionResetIntervalHours` and `sessionResetAnchorTime` (v1 used these for session-anchored cron triggers; v2's cadence model doesn't need them).
5. Add `goalsDir: "~/blitz/goals"`.
6. Bump `version` to `2`.
7. On Windows: replace any existing scheduled tasks with new cadence-based ones. On Mac/Linux: replace cron entries.
8. Migration is automatic and idempotent — re-running it on a v2 config is a no-op.

## SKILL.md framing changes

- **No "100%" or "subscription" or "max out" language.** The v0.1 pivot principle stands.
- Keep the line: *"Parallelism is for throughput, not quota multiplication."*
- New tagline: *"Feed it goals; it ships increments while you're away."*
- The README's "what blitz does" section gets a third bullet: *"Tracks long-term goals on durable plans, ships one increment per fire to a review branch."*

## Hard rules (combined)

| Rule | Enforced where |
|---|---|
| Empty everything = silent exit | M-AUTO step 1 |
| Goal first-fire writes plan only | Goal lifecycle, planning fire |
| Code modifications only on `blitz/*` branches | Goal increment fire step 3 |
| No push to origin without explicit `autoPush: true` | Goal increment fire step 5 |
| Audits never modify project files | Audit fire flow step 7 |
| Per-target idle gate skip = fire defers cleanly | Firing model |
| All work writes artifacts | All fire flows |

## Open questions to resolve in implementation

These are deliberately deferred to the implementation plan, not the spec:

- Exact heuristic for "highest-severity finding" per audit type.
- Whether goal-fire should run the project's tests before committing (likely: yes if test command is detected and runs in <60s, otherwise skip with a warning in the result file).
- How to detect "test command" — `package.json` scripts, `pytest` presence, etc.
- File hashing format for `findingHash` (probably `sha256` over a normalized representation: file path + line + finding type + finding body).

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Goal plan.md drifts from reality after manual edits | Increment fire reads current repo state and verifies the picked item is still applicable; if not, marks it `[~]` (skipped) with a one-line reason and picks the next unchecked. |
| User accumulates 10 stale `blitz/goal-N` branches | `/blitz goal list` shows last-touched date prominently; `/blitz goal remove` does not auto-delete branches but prints a hint. |
| Audit runs become noise | Auto-promote-one-finding rate-limits noise; `/blitz audit disable` removes the project entirely. |
| Cadence fires while the user is in a fast review/coding loop | Per-project git-status idle gate skips. If still too aggressive, blackout windows close the door entirely during work hours. |
| Migration breaks existing v1 user configs | Migration is additive (no destructive renames), idempotent, and version-gated. v1 backlog items keep their IDs and project paths verbatim. |
